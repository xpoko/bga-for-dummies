## Simulation Testing

**One-liner.** Two Studio-only buttons above the turn tracker — **▶ Sim** drives a client-side loop that calls a CheckAction-bypassing server action (`actSimulateStep`) once per turn, picking a random legal move via a pure-PHP picker; **⏭ End** stuffs every empty ice hex with a random penguin and hands off to `RoundEnd` so the framework cascades through scoring. The Sim loop auto-pauses on every `roundEnd` notification so one click = one round-of-water-hex worth of turns. Visible only when `getAllDatas` reports `is_studio=true`, and every action body re-checks `getBgaEnvironment() === 'studio'` so a tampered client cannot drive simulation in production.

## Rulebook → code

There is no rulebook clause for simulation — this entry maps *test-feedback constraints* onto the BGA action model.

> *Constraint set:*
> 1. Driving a full game by clicking through 30+ turns is the only way to inspect end-of-game scoring, Mirror Match half transitions, and the lake-score animation choreography end-to-end. The dev loop must be tight.
> 2. The Studio table only has one human seat by default; the loop must be drivable from any seat (including spectators), not only the active player.
> 3. The simulation surface must not exist in production tables — neither the buttons nor a server-reachable code path.
> 4. The simulator should stress the *placement* code path (the more interesting one) more often than scout, but never violate the rule that the last player on the active track must place if any ice remains.

| Constraint | Implementation |
|---|---|
| 1 — tight feedback loop | Client-side `for` loop ([waddle.ts:348](../src/waddle.ts#L348)) firing `actSimulateStep` per iteration, with auto-pause on `roundEnd` so a 30-turn game collapses to ~5–6 clicks |
| 2 — drivable from any seat | `#[CheckAction(enabled: false)]` on both server actions ([PlayerTurn.php:255-256](../modules/php/States/PlayerTurn.php#L255-L256), [298-299](../modules/php/States/PlayerTurn.php#L298-L299)) — the framework otherwise refuses an action submitted by a non-active player |
| 3 — invisible in production | Two-layer gate: client only renders the buttons when `gamedatas.is_studio` ([waddle.ts:319](../src/waddle.ts#L319)); server-side `assertStudio` ([PlayerTurn.php:369-374](../modules/php/States/PlayerTurn.php#L369-L374)) throws `BgaUserException` on `getBgaEnvironment() !== 'studio'` |
| 4 — weighted random action | `Simulation::pickAction` ([Simulation.php:27-55](../modules/php/Simulation.php#L27-L55)) lists `PLACE_ONE` twice for non-last players (2× weight vs scout) and *only* lists `PLACE_ALL` for the last player when ice is available |

## Design choices

- **Pure picker, not a tangled action method.** All of the decision logic (which actions are legal? which to pick? what weight?) lives in the static `Simulation::pickAction` ([Simulation.php](../modules/php/Simulation.php)). The action method ([PlayerTurn.php:256-294](../modules/php/States/PlayerTurn.php#L256-L294)) only computes the three booleans (`canScout`, `canPlace`, `isLast`), hands them to the picker, and dispatches the returned constant to a helper. This split is what makes the picker unit-testable without a DB or framework — `pickAction(canScout, canPlace, isLast, $randFn)` takes the RNG as an injected callable so tests fix it to a constant index ([SimulationPickerTest.php:71-77](../tests/SimulationPickerTest.php#L71-L77)).
- **Reuse the production helpers, don't fork them.** `simulatePlaceOne` / `simulateScout` / `simulatePlaceAll` ([PlayerTurn.php:384-441](../modules/php/States/PlayerTurn.php#L384-L441)) mirror the bodies of `actPlace` / `actScoutAhead` / `actEndRound` but skip the user-input validation (the picker already picked legal inputs). They still call the same `emitPlacements` / `rotateMarbleToTop` / `notifyAllPlayers('marbleMoved', …)` paths, so a simulated game emits the same notifications as a human-played one. This is what makes the simulation useful for animation debugging — the animations are exactly the ones a real player triggers.
- **One-shot `actSimulateEndGame` for end-game scoring.** Some bugs (the `final_breakdown` rematch bug, the band-grid auto-wrap thresholds) are only visible after `GameEnd` runs. Forcing every empty ice hex to be filled with a random penguin and returning `RoundEnd::class` ([PlayerTurn.php:366](../modules/php/States/PlayerTurn.php#L366)) lets the framework cascade through scoring + (Mirror Match) `MidGameReset` + `GameEnd` without 30 sim clicks. The resulting board is *synthetic* (random colors per hex, no turn-order respect); good for exercising the end-game pipeline, not for inspecting a realistic mid-game snapshot.
- **Auto-pause on `roundEnd`, not on every turn.** A single-step button was tempting (one click = one turn), but a 30-turn game is still 30 clicks, and the inspection moments you actually want are at round boundaries (the active hex marker just moved, a pool may have closed). Hooking the pause request into `notif_roundEnd` ([waddle.ts:1882-1884](../src/waddle.ts#L1882-L1884)) gives one natural break per water-hex completion: 1 click = 1 round. The next click resumes.
- **Two-layer Studio gate (defense in depth).** Just hiding the buttons would mean a player who modifies their client could still POST `actSimulateStep` to a production table — the action method is registered with the framework regardless. The `assertStudio` helper closes that hole; even if the framework dispatches the action, it throws before touching any board state. Belt-and-braces, but the cost is two extra lines per action.
- **`is_studio` lives in `getAllDatas`, not as a build-time flag.** Same JavaScript bundle ships to Studio and production; the only thing that changes is the server response. This means we don't need separate Studio/production builds, and a single CI pipeline covers both. The cost is one boolean in the gamedata payload.
- **Source-text guards instead of integration tests.** Running `actSimulateStep` against a real BGA DB in CI is not feasible (the test harness here is composer/PHPUnit, not a live framework table). Instead [SimulationActionsSourceTest](../tests/SimulationActionsSourceTest.php) reads `PlayerTurn.php` as text and asserts on the *shape* of the action methods: each must contain `assertStudio`, must carry the `#[CheckAction(enabled: false)]` attribute, must call `Simulation::pickAction` (step) or `return RoundEnd::class` (end-game). These are the four regressions that would silently break security or wiring. Same pattern as [PerHexPlacementTest](../tests/PerHexPlacementTest.php).

## Snags & refinements

- **Phantom `$this->checkPossibleAction(...)` call** ([89f1ec3](https://github.com/xpoko/waddle/commit/89f1ec3)). The first version called `$this->checkPossibleAction('actSimulateStep')` to bypass the active-player restriction. That method lives on `GamestateMachine` (per `_ide_helper.php:737`), not on the `GameState` class `PlayerTurn` extends — the call was a no-op at best and a fatal at runtime. The project's own [bga-dev skill docs](#) flag `#[CheckAction(enabled: false)]` as the per-action bypass attribute. Swapped both actions to use the attribute and imported `Bga\GameFramework\Actions\CheckAction`. The source-text guard tests were also rewritten to regex-match the attribute directly preceding each method declaration, so the regression would now be caught at CI time instead of run time.
- **Picker silently 50/50'd the last player into scouting** (also [89f1ec3](https://github.com/xpoko/waddle/commit/89f1ec3)). The original `pickAction` built `[SCOUT, PLACE_ALL]` for `(canScout=true, canPlace=true, isLast=true)` and the rand fell whichever way. `testLastPlayerNeverScoutsEvenWhenBothLegal` documented the intent but the implementation didn't honor it. Fixed by short-circuiting the `isLast` branch: if `canPlace` we *only* offer `PLACE_ALL`; `SCOUT` is now only ever offered as a last-player fallback when no ice is available ([Simulation.php:34-39](../modules/php/Simulation.php#L34-L39)). Added a test for that fallback ([SimulationPickerTest.php:84-96](../tests/SimulationPickerTest.php#L84-L96)).
- **Comment lied about `performAction`'s promise** ([a00c721](https://github.com/xpoko/waddle/commit/a00c721)). The first version of the loop justified its 150 ms `setTimeout` with "wait for the notification queue to drain". That isn't what's happening: `performAction`'s promise resolves on the AJAX response, not on notif dispatch — notifs arrive over a separate channel (per `bga-framework.d.ts:434`). The 150 ms is what gives `notif_roundEnd` time to flip `simPauseRequested` before the next iteration fires the next action. Reworded the comment to honest mechanics so future-us doesn't lower the delay casually.
- **Race window when a notif arrives between iterations** (also [a00c721](https://github.com/xpoko/waddle/commit/a00c721)). The original loop only checked `simPauseRequested` after the 150 ms gap. A late-arriving notif (socket lag, queued behind siblings) could land *between* the previous iteration's post-delay check and the next iteration's `performAction` call — the loop would fire one more turn before noticing. Added a redundant check at the *top* of each iteration ([waddle.ts:356](../src/waddle.ts#L356)) for the cost of one boolean comparison. The post-delay check at the bottom is still load-bearing for the normal case.
- **First pause-trigger was `lakeScored`, not `roundEnd`** ([cce02d7](https://github.com/xpoko/waddle/commit/cce02d7) → [9812c03](https://github.com/xpoko/waddle/commit/9812c03)). The first version paused after each `notif_lakeScored`. That looked right (a pool just scored, natural breakpoint), but lakeScored has its own coverage gaps: multi-water pools whose closure spans rounds don't live-score, they accumulate to `GameEnd`'s batch (see [scoring.md](scoring.md)). So pausing on lakeScored skipped exactly the rounds you'd most want to inspect. Moved the pause request to `notif_roundEnd`, which fires from `RoundEnd::onEnteringState` whenever a water hex's adjacent ice is fully filled — both the regular case (round closed, marker moves to the next hex) and the final case (no more hexes, "All hexes filled — game over"). One pause per water-hex completion is a more predictable rhythm.
- **Mirror Match half-1 → half-2 transition skips a pause.** When `RoundEnd` returns `MidGameReset::class` directly ([RoundEnd.php:50-52](../modules/php/States/RoundEnd.php#L50-L52)) it does so *without* emitting a `roundEnd` notification — the half boundary is its own state. The loop continues seamlessly across the boundary and the next pause comes after the first round of half 2 completes. Acceptable trade-off: the user can always Resume manually after watching the half-2 setup play out.
- **CSS lift only when the controls actually exist.** Production builds don't render `#sim-controls`, but a stylesheet rule for `#track-row { padding-top: 42px }` would still push the tracker down for everyone. The fix is `#track-row:has(#sim-controls)` ([waddle.css:33](../waddle.css#L33-L36)) — applies the lift only when the controls are present in the DOM, so production layout is untouched. The `:has()` selector is now well-supported enough on BGA's target browsers to rely on.
